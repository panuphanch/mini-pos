Handling duplicate-customer rows in the Orders list

Your wife sometimes splits one customer across multiple sheet rows (e.g. K.Ing: one row per cake, but she pays for both). Today each row becomes a separate order with its own receipt and QR. We need a way to combine these at checkout — and to delete obvious duplicates — without breaking the sync model.

The use cases we're solving for

Primary

Same person, split rows on purpose

Wife wrote 2 rows for K.Ing so the kitchen prepares two distinct cakes. K.Ing pays once and wants one receipt with both items.
Happens regularly. This is the case you described.
Edge

Ordered together but separate receipts

A friend orders through K.Ing but wants their own receipt + QR. Two rows that should stay two orders, but be handed over as a pair.
Occasional.
Mistake

Accidental duplicate

Wife typed the same order twice, or a row is empty / wrong. Should just disappear.
Rare but disruptive.

The constraint that shapes everything

The Google Sheet is the source of truth. Sync upserts by (source_tab, source_row). Anything we do in-app to combine or hide rows has to survive the next sync — otherwise the merge will silently un-merge itself when she taps Re-sync. You already have the two primitives we need for this: sync_locked (sync ignores the row) and soft-delete (deleted_at).

Four approaches worth considering

Option A

"Print together" — combine at print time, no DB merge

Cashier ticks 2+ rows in the list, taps Print together. We render one receipt with all line items summed and one QR for the combined total. The orders stay separate in the database; nothing is locked, deleted, or merged.

Option B

Recommended

Option C

Group — link rows without merging them

Select 2+ rows → Group as one payment. The list shows them as a stacked card with a single "Print group" button, but each row is still an independent order under the hood. Best when intent is ambiguous (sometimes one receipt, sometimes two).

Option D

Delete only — soft-delete the unwanted row, edit the other

Add a delete (soft-remove) button to each row, plus the existing Edit order dialog to add the missing item into the surviving order. Doesn't really solve K.Ing on its own; only solves the duplicate-mistake case.
  • Smallest possible change — basically one new button.
  • Soft-delete already exists in the schema.
  • For K.Ing it's a 3-step manual dance: delete row B, open row A, add B's items, save. Easy to mess up.
  • Loses the audit trail of what was actually written on the sheet.

My recommendation

Ship Option B (Merge) + a small Delete action, in that order.
  1. Delete first (½-day): one row, one button, soft-delete + sync_locked. Solves the mistake case immediately and gives you the lock primitive on its own.
  2. Merge next (1–2 days): builds on delete. Pick 2+ rows → master is the lowest source_row → items copy into master → others get deleted_at + sync_locked + merged_into_id = master. Un-merge reads merged_into_id and reverses it.
Option A is tempting for its simplicity, but it leaves the list cluttered and creates a weird split between "displayed order count" and "real order count" that gets worse as more rows pile up over a week.
Sync edge case — the smart way: instead of locking merged-away rows and warning when they change, we tag every line item with the source_row it came from. Sync stays transparent: when a merged row changes in the sheet, the engine replaces the items tagged with that row inside the master and recomputes the total. Wife keeps editing the sheet like normal — no extra steps after the initial merge. sync_locked only fires if she edits the merged order inside the app (discount, delivery fee), at which point sheet edits are ignored as today.

Decisions (2026-05-23)

  1. Different customer names on merge: lenient — allow it, but pop a confirm: "Merge P.Som's order into K.Ing's? She'll get one receipt under K.Ing's name."
  2. Master row deleted from sheet: remove its row-tagged items from master, soft-delete the master if nothing's left. Surface a sync notice.
  3. Discount & delivery fee: sum on merge. If wife wants them split per row she just doesn't merge — the per-row receipt path already works.
  4. Un-merge: defer. Merge has a confirm dialog; accidents should be rare. Add the button later if it becomes a real pain.

What I'd build, concretely

UI changes
  • Add a checkbox column to OrdersPage; selecting 2+ reveals a toolbar with Merge and Delete.
  • Each row gets a trash icon next to the existing edit icon (1 row = quick delete).
  • Merged orders show a merged ×N pill next to the order number. Detail panel lists which source rows contributed which items.
  • "Show removed" checkbox already exists — extend to also surface merged-away rows with a merged into 04 badge and a Restore / un-merge button.
Backend changes
  • Schema: add merged_into_id TEXT NULL REFERENCES "order"(id) on order. Items keep their existing source_row via a new column on order_item (or denormalized on the order).
  • New Tauri commands: orders_delete(id), orders_merge(ids[]), orders_unmerge(id).
  • Sync engine: when iterating sheet rows, skip any with non-null merged_into_id but compare the sheet content against the snapshot; if changed, push a warning into the existing sync-warnings surface.
Testing
  • Merge of 2 rows → one order with combined items, others soft-deleted + locked.
  • Re-sync after merge: no resurrection.
  • Sheet edits a merged-away row → sync produces a warning, does not modify the master.
  • Un-merge → master loses the donated items, donor row comes back, lock cleared.
  • Delete-only path: soft delete + lock, "show removed" reveals it, restore works.
